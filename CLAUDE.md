## Quality gate (must be green before commit / merge)
```bash
uv run pytest -q          # all tests
uv run ruff check .       # lint
uv run mypy src           # type-check (framework source only)
```
`uv` is the package manager — run all tooling via `uv run`. If `uv` is not found, make sure its install directory is on PATH (restart the session after a fresh install).

## Critical conventions
- **`src/framework_cli/template/` is template *payload*, not framework source.** Those `.jinja` / `.py` / config files are rendered into generated projects — do not refactor or lint them as framework code. The framework's own `mypy` excludes that directory. The template is validated by rendering it and exercising the generated project: `tests/test_copier_runner.py` (files render / interpolate) and `tests/acceptance/test_rendered_project.py` (the generated project's own tests, coverage gate, and pre-commit pass).
- Brace-named paths like `src/framework_cli/template/src/{{package_name}}/` are intentional Copier path templating — leave them.
- The CLI (`src/framework_cli/`) is a thin shell over Copier; keep logic in focused modules (`naming.py`, `copier_runner.py`, `cli.py`).
- Changing the template means re-running the render + acceptance tests. A freshly generated project must make a clean first `pre-commit` pass — enforced by `test_rendered_project_precommit_runs_clean`.

## Known follow-ups
- *(resolved on the framework-audit-pass branch, 2026-05-30)* **Gate design quirks — wrong diff + stale findings.** Two issues discovered during the Task 1+1.5+2 commit dance: (1) the gate fed `git diff HEAD~1...HEAD` (prior commit) to reviewers instead of the staged set, so findings were about already-committed content; (2) `_finalize_gate` read ALL records under `findings_dir/` when computing verdict, so prior-run findings persisted into subsequent gate runs. **Fixed:** added `staged_diff()` to `framework_cli.review.diff` (returns `git diff --cached`) and switched `_emit_gate_prep` to use it; `_finalize_gate` now clears stale per-agent records from `findings_dir/` before writing the current run's records (noop/regrade modes leave it alone so the regrade-against-prior-findings workflow still works). A regression test (`test_gate_prepare_diff_is_staged_set_not_head_minus_one`) guards the diff-source fix.
- **Environment-parity reviewer (dev→ci→stage→prod) — a future review agent.** A new reviewer that checks **parity across the environment chain**: a service / env var / exporter / config present in one environment but missing or divergent in another (e.g. in `dev.yml` but not the prod/staging overlay, or an `.env.example` var with no prod wiring). This is exactly the **dev-only-not-prod** class that bit OBS-PROD (obs stack dev-only) and SVC-PROD (battery services dev-only) — both caught by a **human**, not an agent. Natural fit: an **agentic** reviewer (parity spans `infra/compose/*.yml` overlays + `.env.example` + env configs, so it must follow references across files). Complements the observability-completeness check below (that one is §5-contract-per-surface; this one is cross-environment consistency). Schedule as its own small slice after the context-aware redesign lands. [Noted 2026-05-29.]
- **Observability-completeness review gap — we want a review check that enforces the §5 contract.** The existing `review-observability` agent is **diff/code-level** (untraced code paths, unlogged errors, missing SLO thresholds — it reviews a generated project's app diffs). It does NOT catch the class of gap that has now slipped twice and been caught by a **human**, not an agent: *a new runtime surface / service / store added WITHOUT its §5 observability* (metrics-or-exporter + scrape target + alert + dashboard + `/health` signal). The mongodb battery shipped full obs while the baseline postgres store had none (→ Plan 8i); future batteries/services could regress the same way. **Want:** either a new review agent (e.g. `review-observability-completeness`) or a framework-side test/lint that flags "a battery/service that adds a compose service or external store but no scrape target + alert + dashboard." Note this is partly a **framework-authoring** check (on us), distinct from the generated-project diff review the current agents do — so it may be a `tests/` invariant (every service-adding battery has an alert+dashboard) rather than (or in addition to) an LLM agent. Decide the form when scheduling it (candidate: alongside Plan 9 dogfooding, or its own small slice). **Also (from the OBS-PROD work):** (a) the check should include **observability-infrastructure scaling** — the OBS-PROD co-located per-host obs stack is a *starting point*; the reviewer/check should flag when it needs to graduate (centralize the stack, dedicated obs host, retention/resource sizing as the app grows) rather than silently outgrowing a single host; (b) **consider splitting `review-observability` into domain reviewers — infra / db / app / fe** (the current agent is app/code-level only; OBS-PROD is *infrastructure* obs, db batteries are *data* obs, 8g/react brings *frontend* obs) — one coarse agent likely under-serves all four domains.
- *(resolved by SVC-PROD)* **Battery services (mongo/redis) + their exporters weren't in prod/staging.** A `mongodb`/`workers` project had no `mongo`/`redis`/`worker`/`beat` (nor exporters) in prod — same dev-only-not-prod class as OBS-PROD, but *functional*. Fixed by the prod/staging-only **`infra/compose/services.yml` overlay** (gated mongo/redis + worker/beat on `${APP_IMAGE}` with `APP_RUN_MIGRATIONS:false`, merged by `strategy.sh` alongside `observability.yml`) + relocating `mongodb-exporter`/`celery-exporter` into `observability.yml` (battery metrics now run in prod). **Remaining sub-gap for slice 2:** each *new* service-adding battery (timescaledb/neo4j/redis) must add its service to `services.yml` (and exporter to `observability.yml`) — the obs-completeness check below should enforce this for prod, not just dev.
- **Docker acceptance tier fills `/tmp` and can wedge the sandbox.** The `tests/acceptance/` tier renders many projects under `/tmp/pytest-of-chris/` and runs containers as **root**, leaving root-owned `__pycache__` dirs that pytest can't clean → they accumulate and fill `/tmp`. Because the bash tool also writes temp files to `/tmp`, a full `/tmp` makes **every** shell command fail (exit 1, no output) — the sandbox goes non-functional and can't self-recover (clearing `/tmp` needs `/tmp`). This wedged SVC-PROD Task 4 (the user cleared it from a host terminal with `sudo rm -rf /tmp/pytest-of-chris/*`). **Fix candidates:** run the acceptance containers as the host UID (e.g. `--user $(id -u):$(id -g)` / Compose `user:`), or have the tests clean up their renders (incl. root-owned artifacts) in teardown, or point pytest's basetemp at a roomier disk. Until fixed: avoid running the full acceptance tier in-session; prefer `--ignore=tests/acceptance` + the fast `docker compose config` merge-validations, and `rm -rf /tmp/pytest-of-chris/* 2>/dev/null` (sudo if needed) after any acceptance run. **Now an explicit Plan 9 deliverable** (run acceptance containers as the host UID + clean up renders in teardown — see the Plan 9 row).
