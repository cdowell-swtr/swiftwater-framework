## Critical conventions
- **`src/framework_cli/template/` is template *payload*, not framework source.** Those `.jinja` / `.py` / config files are rendered into generated projects ‚Äî do not refactor or lint them as framework code. The framework's own `mypy` excludes that directory. The template is validated by rendering it and exercising the generated project: `tests/test_copier_runner.py` (files render / interpolate) and `tests/acceptance/test_rendered_project.py` (the generated project's own tests, coverage gate, and pre-commit pass).
- Brace-named paths like `src/framework_cli/template/src/{{package_name}}/` are intentional Copier path templating ‚Äî leave them.
- The CLI (`src/framework_cli/`) is a thin shell over Copier; keep logic in focused modules (`naming.py`, `copier_runner.py`, `cli.py`).
- Changing the template means re-running the render + acceptance tests. A freshly generated project must make a clean first `pre-commit` pass ‚Äî enforced by `test_rendered_project_precommit_runs_clean`.

## Known follow-ups
- *(resolved on the framework-audit-pass branch, 2026-05-30)* **Gate design quirks ‚Äî wrong diff + stale findings.** Two issues discovered during the Task 1+1.5+2 commit dance: (1) the gate fed `git diff HEAD~1...HEAD` (prior commit) to reviewers instead of the staged set, so findings were about already-committed content; (2) `_finalize_gate` read ALL records under `findings_dir/` when computing verdict, so prior-run findings persisted into subsequent gate runs. **Fixed:** added `staged_diff()` to `framework_cli.review.diff` (returns `git diff --cached`) and switched `_emit_gate_prep` to use it; `_finalize_gate` now clears stale per-agent records from `findings_dir/` before writing the current run's records (noop/regrade modes leave it alone so the regrade-against-prior-findings workflow still works). A regression test (`test_gate_prepare_diff_is_staged_set_not_head_minus_one`) guards the diff-source fix.
- **Environment-parity reviewer (dev‚Üíci‚Üístage‚Üíprod) ‚Äî a future review agent.** A new reviewer that checks **parity across the environment chain**: a service / env var / exporter / config present in one environment but missing or divergent in another (e.g. in `dev.yml` but not the prod/staging overlay, or an `.env.example` var with no prod wiring). This is exactly the **dev-only-not-prod** class that bit OBS-PROD (obs stack dev-only) and SVC-PROD (battery services dev-only) ‚Äî both caught by a **human**, not an agent. Natural fit: an **agentic** reviewer (parity spans `infra/compose/*.yml` overlays + `.env.example` + env configs, so it must follow references across files). Complements the observability-completeness check below (that one is ¬ß5-contract-per-surface; this one is cross-environment consistency). Schedule as its own small slice after the context-aware redesign lands. [Noted 2026-05-29.]
- **Observability-completeness review gap ‚Äî we want a review check that enforces the ¬ß5 contract.** The existing `review-observability` agent is **diff/code-level** (untraced code paths, unlogged errors, missing SLO thresholds ‚Äî it reviews a generated project's app diffs). It does NOT catch the class of gap that has now slipped twice and been caught by a **human**, not an agent: *a new runtime surface / service / store added WITHOUT its ¬ß5 observability* (metrics-or-exporter + scrape target + alert + dashboard + `/health` signal). The mongodb battery shipped full obs while the baseline postgres store had none (‚Üí Plan 8i); future batteries/services could regress the same way. **Want:** either a new review agent (e.g. `review-observability-completeness`) or a framework-side test/lint that flags "a battery/service that adds a compose service or external store but no scrape target + alert + dashboard." Note this is partly a **framework-authoring** check (on us), distinct from the generated-project diff review the current agents do ‚Äî so it may be a `tests/` invariant (every service-adding battery has an alert+dashboard) rather than (or in addition to) an LLM agent. Decide the form when scheduling it (candidate: alongside Plan 9 dogfooding, or its own small slice). **Also (from the OBS-PROD work):** (a) the check should include **observability-infrastructure scaling** ‚Äî the OBS-PROD co-located per-host obs stack is a *starting point*; the reviewer/check should flag when it needs to graduate (centralize the stack, dedicated obs host, retention/resource sizing as the app grows) rather than silently outgrowing a single host; (b) **consider splitting `review-observability` into domain reviewers ‚Äî infra / db / app / fe** (the current agent is app/code-level only; OBS-PROD is *infrastructure* obs, db batteries are *data* obs, 8g/react brings *frontend* obs) ‚Äî one coarse agent likely under-serves all four domains.
