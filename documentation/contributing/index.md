# Contributing

This page is for people working on the **framework itself** — not on a project generated by the framework.

---

## Dev setup

Clone the repository and install dependencies:

```bash
git clone https://github.com/cdowell-swtr/swiftwater-framework
cd swiftwater-framework
uv sync
```

`uv` is the package manager for all tooling. If `uv` is not found, install it and restart your shell.

---

## Quality gate

The following must be green before every commit and merge:

```bash
uv run pytest -q          # all tests
uv run ruff check .       # lint
uv run mypy src           # type-check (framework source only)
```

The pre-commit hook enforces this at commit time. Run the commands manually before committing to catch failures early.

---

## Building and serving these docs

Live preview (auto-reloads on save):

```bash
uv run --group docs mkdocs serve
```

Production build (strict mode — the same check the docs CI runs):

```bash
uv run --group docs mkdocs build --strict
```

The build must exit 0 with no warnings before a docs change is merged.

## Where the design lives

All design documents live under `docs/superpowers/`:

- **Specs** — `docs/superpowers/specs/` — feature and subsystem design documents.
- **Plans** — `docs/superpowers/plans/` — implementation plans and their status.
- **Build roadmap** — `docs/superpowers/plans/2026-05-20-meta-plan.md` — the authoritative record of what is done, what is in flight, and what is next.

`CLAUDE.md` at the repository root is the working agreement: conventions, quality gate, model policy, and current state. It is the first thing to read when picking up work in this repository.

---

## Dogfooding

The framework holds itself to the same discipline it imposes on generated projects: the same gates (pytest, ruff, mypy), the same CI structure, and its own review agents running against every pull request. A framework that exempts itself from its own standards cannot be trusted to enforce them.
