The repeatable release-cut sequence for this repo (done identically for v0.1.5/v0.1.6/v0.1.7/v0.1.8/v0.1.9). **Trigger:** cut a release whenever something that reaches builders ships — NOT only template-payload changes. Framework-CLI features ship in the wheel too: e.g. **v0.1.9 was a framework-side-only change** (the new `review-env-parity` reviewer — a prompt + registry entry, zero template files touched) and still warranted a release because the reviewer reaches builders' CI via the installed `framework-cli` wheel. When in doubt, release.

1. **Bump version** in `pyproject.toml` (`version = "X.Y.Z"`).
2. `uv lock` (updates `framework-cli` to X.Y.Z in `uv.lock`).
3. **`DOGFOOD_COMMIT`** in `src/framework_cli/dogfood.py` → `"vX.Y.Z"` (the tag the generated integrity step-0 installs).
4. Update **PLAN.md** — mark the shipped FWK row(s) `[x] SHIPPED vX.Y.Z` and append an `ACTION_LOG.md` release entry. (The **meta-plan is FROZEN** through v0.2.4 — do NOT update it for recent releases; PLAN.md + ACTION_LOG.md are the live status. `CLAUDE.md` needs no per-release bump.)
5. Validate metadata: `uv run ruff check .`, `uv run python -m mypy src`, `uv lock --check`, `uv build` (must build `framework_cli-X.Y.Z.{whl,tar.gz}`), AND the version-consistency suite `uv run python -m pytest tests/test_release.py tests/test_cli.py` (asserts `pyproject` == `DOGFOOD_COMMIT` == `uv.lock`; ~136 tests).
6. **The release rides a PR — the default branch (`main`) is PROTECTED, no direct pushes** (ruleset 17579429: PR required, required checks `gate`+`build`+`render-complete`, self-merge OK). Branch `release/vX.Y.Z` off `main`; `git add` then `git commit -m "chore(release): vX.Y.Z — …"` as **SEPARATE** Bash calls (commit-gate). The old `framework gate-prepare` skip-marker dance is **RETIRED** — the PreToolUse hook self-runs `framework gate` skip-neutral (no `review.toml` → trivial PASS); just stage `PLAN.md`/`ACTION_LOG.md` ([[controller-skip-marker-recipe]]). Keep the word "commit" out of Bash *descriptions* too ([[commit-gate-hook-false-matches-git-commit-substring]]).
7. Push `release/vX.Y.Z`, open the **release PR** to `main` (`gh pr create --base main`). Its `render-matrix.yml` run is the framework-side proof — watch it green (`gh pr checks <#> --watch`). **If `main` advanced since you branched, the ruleset blocks the merge ("head branch not up to date")** → `git merge origin/main` into the release branch (NOT rebase — preserve any commit SHAs your scorecard/ACTION_LOG cite), push, and CI **re-runs** (~12 min, the single unsharded `gate` job is the long pole). The repo has **no auto-merge** (`enablePullRequestAutoMerge` off) → you must merge manually once green.
8. Once `gate`+`build`+`render-complete` are green and the PR is `MERGEABLE`/`CLEAN`, **squash-merge** to `main` (`gh pr merge <#> --squash --delete-branch`, matching the repo convention #86/#87/#88/#89) and record the **merge commit SHA** (`gh pr view <#> --json mergeCommit`). Sync local `main` (`git checkout main && git pull --ff-only`).
9. **Lightweight tag on the merge commit** (`git tag vX.Y.Z <merge-sha>`, NOT annotated — prior tags are lightweight) + `git push origin vX.Y.Z` → triggers `release.yml` (tag==version guard → reusable ci + render-matrix → `uv build` → GitHub Release). Watch it (`gh run watch <id> --exit-status`) to a published Release (`gh release view vX.Y.Z`: `isDraft:false`, `isPrerelease:false`, **2 assets** `framework_cli-X.Y.Z-py3-none-any.whl` + `.tar.gz`).

**Why:** the local gate misses ruff-FORMAT, template-payload mypy, and generated-project dep drift ([[release-readiness-needs-render-not-local-gate]]) — the render-matrix on the release PR is the real proof, and `release.yml` re-runs it on the tag. **How to apply:** the version-bump can no longer be a direct push to `master` (memory pre-dates the branch-protection ruleset + the `master`→`main` rename) — it goes through a release PR; the tag (created AFTER merge, on the merge commit) must equal `pyproject` version or `release.py`'s guard fails. `DOGFOOD_COMMIT = "vX.Y.Z"` legitimately points at a not-yet-existing tag during the PR — the render-matrix does not require the tag to pre-exist (it renders `_commit: vX.Y.Z`; the tag lands seconds after merge). Verified end-to-end for **v0.4.3** (combined SP2+SP3): release PR #89 → squash-merge `d585bcc` → tag `v0.4.3` → published Release.
