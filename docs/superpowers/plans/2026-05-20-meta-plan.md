## Proposed Sub-Slicing for Plan 3 (largest item)

Plan 3 bundles several subsystems; when written it should be split so each is buildable and testable on its own:

- **3a — Local runtime:** Docker Compose `base`/`dev`/`dev:lite`/`test` profiles, Traefik + mkcert local HTTPS, hot reload, container startup ordering (healthchecks + `depends_on: service_healthy`), `.gitattributes`/TZ=UTC consistency, `SERVICES.md`. Plus structlog + correlation IDs and a *real* SLO-evaluating `/health` (replacing the walking-skeleton stub). Testable: `task dev` brings up the app over HTTPS; `/health` returns live SLO status.
- **3b — Observability stack:** Prometheus, Grafana (auto-provisioned dashboards from SLO defs), Loki + Promtail, Tempo (traces), Alertmanager, OTEL Collector — wired so a dev sees their dashboard locally. Alert channels come from the `framework new` wizard.
- **3c — Database lifecycle:** the `database` paradigm wizard's relational path (PostgreSQL + SQLAlchemy + Alembic), DB init/migrate/seed on first `task dev`, isolated test DB in the `test` profile. (Other paradigms / the full wizard may fold into Plan 8's `database` battery.)

This split is a proposal to confirm at the start of Plan 3, not a commitment.

## Carried-Forward Notes (address in the relevant plan)

Harmless now (no update flow), but **Plan 6** must address it: `copier update`/`framework upskill` needs a portable, versioned template source (the template is currently bundled in the package and rendered from a local path; `copier update` wants a git repo with tags/commits). Decide the template-source/versioning model in Plan 6.
- **Coverage suite-type contexts** (unit vs functional vs e2e as separate labels) — **delivered in Plan 5a** via `scripts/coverage.sh` (`coverage run --context=<suite>`); the e2e tier now exists, the pre-commit gate is unit+functional ≥70% and the CI gate is unit+functional+e2e ≥85%. (Plan 2 had set up per-test dynamic contexts + a 70% threshold only.)
- **Concrete deploy strategy** (spec §21) is still an open decision — no target chosen. The deploy *contract* is specified; **Plan 5b** wires the seam (Plan 5a's `ci.yml` ships only the integrity/review seams, not a deploy job), and a first concrete strategy (compose-over-SSH / a PaaS / k8s) is chosen later.
- **Full multi-file-type linters** (hadolint, actionlint, shellcheck, eslint, stylelint, prettier) are added with the plans/batteries that introduce those file types (Dockerfiles → 3, workflows → 5, React → 8). Remaining §5 linters (taplo, yamllint, hadolint, prettier) — complete the linter set in a tidy-up plan; actionlint+shellcheck landed in 5a.
- **Other Layer-2 hooks** (test-file→pytest-on-save, other file types, pip-audit, gitleaks-on-`.env`) deferred from 2b; revisit when the relevant tooling/files exist.

## Environment Notes (for resuming)
- **uv** must be installed and on `PATH` (`uv --version` works). If you just installed it, restart the session/terminal so the updated `PATH` is picked up — a session started before the install has a stale `PATH`. Install: https://docs.astral.sh/uv/getting-started/installation/
- **Docker** is installed in the dev environment (native `docker.io` 29.1.3 + `docker-compose-v2`, running under systemd in WSL2). The Docker-gated **Layer-B** acceptance tests in `tests/acceptance/test_rendered_project.py` — the rendered project's testcontainers-Postgres suite and the live `docker compose` stack tests — now run locally (they `skipif(not _docker_available())` otherwise). On a fresh machine without Docker, install it to run these; the generated projects' own DB tests likewise require Docker (testcontainers, by design — no SQLite fallback).
- The framework's own quality gate: `uv run pytest -q`, `uv run ruff check .`, `uv run mypy src`.
