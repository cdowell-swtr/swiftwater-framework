## Sub-Slicing for Plan 8 (batteries) â€” confirmed 2026-05-24

The battery *mechanism* first, then one sub-plan per battery. Each sub-plan still gets its own spec â†’ plan â†’ implementation cycle.

- **Relational (PostgreSQL) stays always-on, not a battery.** It is the framework's prescribed relational spine (offload architecture *from* the builder; Â§1's "wrong paradigm" failure). Retrofitting it into a removable battery was considered and **rejected**: the one real benefit â€” grounding the `review-data-integrity`/`review-data-lineage` agents in the project's store set â€” comes from *declaring the paradigm set* (recorded in `.copier-answers.yml` and fed to those agents as context), **not** from making relational removable; the retrofit's cost (conditional-rendering churn, double-variant acceptance testing, eroding the prescription, front-loading the hardest refactor) outweighs it. The tempting middle (a default-on *non-removable* relational battery) is also rejected â€” it pays the refactor cost for none of the removability upside. Revisit only if a concrete no-relational demand appears or the agents prove they need relational modeled as a uniform battery; a `--without database` escape hatch is the lighter option if so.
- **Two operations, asymmetric risk.** `--with` (add) is additive and safe. `--downskill`/`--without` (remove) is not: Copier's `run_update` only moves forward, so removal needs framework-owned logic (the answers file + integrity manifest as the removal map) plus a heuristic **usage-detection** scan (builder code/config still referencing the battery â†’ warn/block unless `--force`), then a post-removal integrity + test re-run. Hence **8a-1 (additive)** vs **8a-2 (removal)**, with 8a-2 sequenced after â‰Ą1 battery exists to test against.
- **Battery dependencies/implications** are a first-class mechanism concern (pgvector/timescaledb â‡’ postgres); both direct `--with` and the wizard resolve a consistent set.
- **Database paradigms = per-paradigm atomic batteries + a wizard front-end.** Experts run `--with redis|mongodb|pgvector|timescaledb|neo4j` directly; `--with database` is the guided interview that *resolves to the same atomic batteries* (no duplicate scaffolding). Per-paradigm `--downskill` is exactly the "drop the graph DB" case. **The wizard (`8f-w`) is unified/configurable â€” it also covers alert-channel selection** (Slack/email/PagerDuty â†’ alertmanager receivers + `.env.example` secrets; the OBS-PROD slice ships only a minimal secret-driven webhook, the wizard adds the multi-channel ergonomics). Both are scaffold-time config, built together, last in Plan 8.
- **The 3 deferred battery review agents land with their gating battery** â€” `review-api-design` with 8d (graphql/REST), `review-accessibility` + `review-usability` with 8g (react) â€” and are auto-evaluated by the Plan 7d harness once registered + given fixtures.
- **Sequencing:** 8a-1 (mechanism, proven with a small vehicle battery) â†’ self-contained batteries (8b webhooks, 8c workers [+ Plan 4 DLQ], 8e websockets) â†’ 8a-2 (removal) once there's something to remove â†’ 8d graphql â†’ 8f database paradigms + wizard â†’ 8g react â†’ 8h consumers. **Start with 8a-1.**
