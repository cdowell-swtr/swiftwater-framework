## Sub-Slicing for Plan 8 (batteries) ‚Äî confirmed 2026-05-24

The battery *mechanism* first, then one sub-plan per battery. Each sub-plan still gets its own spec ‚Üí plan ‚Üí implementation cycle.

- **Relational (PostgreSQL) stays always-on, not a battery.** It is the framework's prescribed relational spine (offload architecture *from* the builder; ¬ß1's "wrong paradigm" failure). Retrofitting it into a removable battery was considered and **rejected**: the one real benefit ‚Äî grounding the `review-data-integrity`/`review-data-lineage` agents in the project's store set ‚Äî comes from *declaring the paradigm set* (recorded in `.copier-answers.yml` and fed to those agents as context), **not** from making relational removable; the retrofit's cost (conditional-rendering churn, double-variant acceptance testing, eroding the prescription, front-loading the hardest refactor) outweighs it. The tempting middle (a default-on *non-removable* relational battery) is also rejected ‚Äî it pays the refactor cost for none of the removability upside. Revisit only if a concrete no-relational demand appears or the agents prove they need relational modeled as a uniform battery; a `--without database` escape hatch is the lighter option if so.
- **Two operations, asymmetric risk.** `--with` (add) is additive and safe. `--downskill`/`--without` (remove) is not: Copier's `run_update` only moves forward, so removal needs framework-owned logic (the answers file + integrity manifest as the removal map) plus a heuristic **usage-detection** scan (builder code/config still referencing the battery ‚Üí warn/block unless `--force`), then a post-removal integrity + test re-run. Hence **8a-1 (additive)** vs **8a-2 (removal)**, with 8a-2 sequenced after ‚â•1 battery exists to test against.
- **Battery dependencies/implications** are a first-class mechanism concern (pgvector/timescaledb ‚áí postgres); both direct `--with` and the wizard resolve a consistent set.
- **Database paradigms = per-paradigm atomic batteries + a wizard front-end.** Experts run `--with redis|mongodb|pgvector|timescaledb|neo4j` directly; `--with database` is the guided interview that *resolves to the same atomic batteries* (no duplicate scaffolding). Per-paradigm `--downskill` is exactly the "drop the graph DB" case.
- **The 3 deferred battery review agents land with their gating battery** ‚Äî `review-api-design` with 8d (graphql/REST), `review-accessibility` + `review-usability` with 8g (react) ‚Äî and are auto-evaluated by the Plan 7d harness once registered + given fixtures.
- **Sequencing:** 8a-1 (mechanism, proven with a small vehicle battery) ‚Üí self-contained batteries (8b webhooks, 8c workers [+ Plan 4 DLQ], 8e websockets) ‚Üí 8a-2 (removal) once there's something to remove ‚Üí 8d graphql ‚Üí 8f database paradigms + wizard ‚Üí 8g react ‚Üí 8h consumers. **Start with 8a-1.**
